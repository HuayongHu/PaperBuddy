## 一、项目概述

### 1.1 项目定位

一款面向**大学生、研究生、博士生**的学术写作辅助 Web 应用，提供两大核心功能：

- **论文润色**：用户粘贴论文片段，AI 逐段给出问题分析与修改建议
- **文献解读**：用户上传 PDF 论文，AI 基于全文内容进行多轮智能问答

### 1.2 核心设计原则

- **无需注册、无需填写密钥**：API Key 由部署者通过 Vercel 环境变量配置，对用户完全透明
- **开箱即用**：用户访问即可使用，零门槛
- **界面简洁克制**：参照 Claude.ai 风格，以内容为中心，无花哨装饰
- **纯中文界面**：所有 UI 文字、提示信息、错误提示均使用中文

### 1.3 产品边界（明确不包含的功能）

- 不需要用户登录/注册系统
- 不需要服务端数据库（历史记录仅存 localStorage）
- 不需要用户自定义 API Key
- 不需要多用户管理、权限控制
- 不需要支持除 PDF 外的其他文件格式（如 Word、LaTeX）
- 不需要移动端 Native App

### 2.2 论文润色模块

#### 2.2.1 路由
`/polish`

#### 2.2.2 整体布局（桌面端，≥1024px）

│  │                      │  │  └───────────────────────────┘  │
│  │  文本输入框           │  │                                   │
│  │  （可拖拽调整高度）    │  │  [一键复制全文润色结果]           │

│  └──────────────────────┘  │                                   │
│  字数统计 / 语言检测提示     │                                   │
│  [清空]        [提交润色 →] │                                   │
│                          │                                   │
│  ──── 润色历史 ────        │                                   │
│  • 2024-12-01 14:23 学术加强│                                   │
│  • 2024-11-30 09:11 轻度润色│                                   │
│  （最近 10 条，可点击恢复）  │                                   │
└──────────────────────────┴───────────────────────────────────┘
```

#### 2.2.3 润色模式说明

| 模式 ID | 显示名 | 说明 |
|---|---|---|
| `light` | 轻度润色 | 保留原文结构与表达意图，仅修正语法错误、病句、冗余表达、用词不当 |
| `academic` | 学术加强 | 在轻度润色基础上，进一步提升学术语气、规范专业术语用法、强化逻辑连贯性 |
| `rewrite` | 结构重写 | 在保留核心论点与数据的前提下，大幅重组行文逻辑、改写句式结构 |

- 默认选中：`light`（轻度润色）
- 以标签页（Tab）或分段按钮（Segmented Control）形式展示，选中态用主色调背景+白字

#### 2.2.4 文本输入区

- **组件类型**：`<textarea>`，允许用户拖拽改变高度（`resize: vertical`）
- **默认高度**：约 12 行
- **占位提示文字**（placeholder）：
  ```
  在此粘贴需要润色的论文片段（段落或章节）…
  
  支持中英文论文，建议每次提交不超过 2000 字。
  超过 2000 字时，系统将提示您分段提交。
  ```
- **字数统计**：输入框下方实时显示 `当前字数: X 字`，超过 2000 字时变红并显示警告
- **语言自动检测**：提交时前端检测输入文字的主体语言（中文/英文），并在请求中带上 `language` 字段供后端提示词使用
- **清空按钮**：点击弹出 `confirm` 确认后清除输入框内容及右侧结果区
- **提交按钮**：
  - 正常态：`提交润色 →`，主色调填充背景
  - Loading 态：按钮禁用，文字变为 `润色中…`，显示旋转 spinner
  - 字数超限时：按钮变灰禁用，显示 `文字超出限制`

#### 2.2.5 润色结果展示（批注卡片样式）

**结果格式说明：**

后端 API 以流式（SSE）方式返回文本，模型按照以下格式逐段输出内容（详见第六章 System Prompt）：

```
---PARA_START---
【原文】
（原始段落内容）

【问题分析】
- 问题1：xxx
- 问题2：xxx

【润色结果】
（润色后的段落内容）
---PARA_END---
```

**前端解析逻辑：**

- 监听流式数据，通过 `---PARA_START---` 和 `---PARA_END---` 定界符识别段落块
- 每接收完一个完整的段落块，立即在右侧渲染一张「批注卡片」
- 段落块未接收完时，显示该卡片的骨架屏（skeleton）占位
- 渲染顺序从上到下，和原文段落顺序对应

**批注卡片 UI 结构（单卡片）：**

```
┌────────────────────────────────────────────┐
│ 段落 1                              [收起 ∧] │
├────────────────────────────────────────────┤
│ 📄 原文                                     │
│ ┌──────────────────────────────────────┐   │
│ │ （灰色背景，原始段落文字）             │   │
│ └──────────────────────────────────────┘   │
├────────────────────────────────────────────┤
│ ⚠ 问题分析                                  │
│  • 问题1：xxx                               │
│  • 问题2：xxx                               │
├────────────────────────────────────────────┤
│ ✅ 润色结果                      [复制本段] │
│ ┌──────────────────────────────────────┐   │
│ │ （浅绿背景，润色后段落文字）           │   │
│ └──────────────────────────────────────┘   │
└────────────────────────────────────────────┘
```

**配色说明：**
- 原文区块背景：`#F5F4EF`（暖灰），文字颜色 `#555555`
- 问题分析标题：`#DC2626`（红色），列表项文字颜色 `#374151`
- 润色结果区块背景：`#F0FDF4`（浅绿），文字颜色 `#1A1A1A`
- 卡片整体：白色背景，`1px` 边框（`#E5E5E3`），`8px` 圆角，轻阴影

**「一键复制全文润色结果」按钮：**
- 位于右侧面板底部（在所有卡片的下方）
- 点击时将所有「润色结果」区块的文字按段落顺序合并，中间以空行分隔，写入剪贴板
- 复制成功后按钮文字短暂变为「✓ 已复制」（1.5 秒后恢复）

**结果区初始状态（未提交时）：**
```
（居中显示，淡灰色文字）
📝 提交论文片段后，润色建议将逐段显示在这里
```

#### 2.2.6 润色历史记录（左侧面板底部）

- 以折叠区域展示，默认折叠
- 展开后显示最近 10 条记录，每条信息：`时间 + 润色模式 + 前20字预览`
- 点击某条历史记录：将该次的输入文本和润色模式恢复到输入区，右侧结果区恢复该次结果
- 每条历史记录旁有「删除」图标（×），点击后从 localStorage 删除

### 2.3 PDF 文献解读模块

#### 2.3.1 路由
`/pdf`

#### 2.3.2 上传前状态（Upload State）

首次进入 `/pdf` 路由，或没有已载入 PDF 时，显示上传界面：

```
┌─────────────────────────────────────────────────────────┐
│  导航栏                                                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│                    （垂直水平居中）                       │
│                                                         │
│           ┌─────────────────────────────┐               │
│           │                             │               │
│           │       📄                    │               │
│           │                             │               │
│           │  拖拽 PDF 文件到此处          │               │
│           │  或点击选择文件              │               │
│           │                             │               │
│           │  支持 PDF 格式，文件大小 ≤ 20MB│               │
│           │                             │               │
│           └─────────────────────────────┘               │
│                                                         │
│           [或从历史记录中选择之前上传的文献]              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**上传区域交互：**
- 拖入文件时，虚线边框变为实线 + 背景色加深（hover 态）
- 点击区域触发文件选择对话框（`accept=".pdf"`）
- 文件选中后立即开始处理，无需再点确认

**文件处理步骤（前端）：**

1. 验证文件格式（`file.type === 'application/pdf'`）和大小（≤ 20MB）
2. 如验证失败，在上传区域显示红色错误提示
3. 验证通过后，显示进度条与提示文字：`「正在提取文本内容，请稍候…」`
4. 使用 `pdfjs-dist` 完成两件事（并行）：
   - **渲染**：将 PDF 以页面形式渲染到左侧预览区（canvas 方式）
   - **文本提取**：提取所有页面的文字内容，拼接为长字符串
5. 文本提取完成后，进入「对话状态」

**文本提取注意事项：**
- 提取完成后，记录提取到的总字符数
- 如提取到的文字为空（可能是扫描版 PDF），显示警告：「该 PDF 为扫描版或图片型，无法提取文字内容，问答功能可能无法正常工作。」
- 将提取的文本截断至最多 **60,000 个字符**（约 4 万汉字或 1 万英文单词），超出部分截断并提示用户：「文献内容较长，已自动截取前 60,000 字符进行分析。」

#### 2.3.3 对话状态布局（桌面端，≥1024px）

┌──────────────────────────────────────────────────────────┐
│  导航栏  ←返回 ｜ [当前文件名.pdf] ｜ [重新上传]            │
├────────────────────────┬─────────────────────────────────┤
│  左侧：PDF 预览（45%）  │  右侧：智能对话（55%）            │

│  │   PDF 原文渲染   │  │  │                           │  │
│  │   （可滚动）     │  │  │  快捷提问按钮：             │  │
│  │                  │  │  │  [概述主要贡献]             │  │

#### 2.3.4 PDF 预览区

- 使用 `pdfjs-dist` 将每页渲染为 `<canvas>` 元素
- 默认展示第 1 页
- 提供「上一页」/「下一页」按钮和页码显示（`第 X / N 页`）
- 支持在预览区内滚动（overflow-y: auto）
- 预览区宽度固定，canvas 宽度自适应预览区宽度（等比缩放高度）

#### 2.3.5 智能对话区

**AI 欢迎消息（PDF 加载成功后自动显示）：**
```
我已读取《[文件名]》的全文内容（共 X 页，约 X 字）。

您可以直接向我提问，例如：
• 论文的核心研究问题是什么？
• 作者使用了哪些实验方法？
• 论文得出了哪些结论？
```

**快捷提问按钮（显示在欢迎消息下方）：**
- 「概述主要贡献」
- 「解释研究方法」
- 「实验结果如何？」
- 「本文创新点是什么？」
- 点击任一按钮，视为用户输入该问题并立即发送
- 第一次发送后，快捷按钮隐藏（避免反复触发）

**消息气泡样式：**

| 角色 | 对齐 | 背景色 | 文字色 | 圆角 |
|---|---|---|---|---|
| 用户 | 右对齐 | `#1A1A1A`（深色） | `#FFFFFF` | `16px 16px 4px 16px` |
| AI | 左对齐 | `#F5F4EF`（暖灰） | `#1A1A1A` | `16px 16px 16px 4px` |

**AI 消息附加操作：**
- 每条 AI 回复消息右下角有「复制」图标按钮
- 复制后显示「✓」确认图标（1 秒后恢复）

**Markdown 渲染：**
- 使用 `react-markdown` + `remark-gfm` 渲染 AI 回复
- 支持：标题（h1-h4）、加粗、斜体、无序/有序列表、代码块、引用块、表格
- 代码块使用 `monospace` 字体，背景色 `#F3F4F6`

**流式输出：**
- AI 回复消息在接收到第一个 token 后立即在 UI 中出现
- 逐字符/逐词追加显示，产生「实时打字」效果
- 流式接收过程中，消息末尾显示闪烁光标 `▋`
- 接收完成后光标消失

**输入框区域：**
- `<textarea>` 支持多行输入（自动扩展高度，最高约 6 行）
- Enter 发送，Shift+Enter 换行
- AI 响应过程中，输入框禁用，发送按钮变为「停止」按钮（点击可中断流式输出）
- 空内容时发送按钮禁用

#### 2.3.6 历史对话恢复

- 对话记录以 PDF 文件名（`encodeURIComponent(filename)`）为 key 存储于 localStorage
- 重新上传**同名 PDF** 时，弹出确认提示：
  ```
  检测到该文献的历史对话记录（共 X 条消息）
  [恢复历史对话]  [开始新对话]
  ```
- 选择「恢复」时，加载历史消息到对话区，然后可继续对话
- 选择「开始新对话」时，清空并从欢迎消息重新开始

#### 2.3.7 历史文献记录（上传前页面底部）

- 展示用户此前在本浏览器上传过的文献（从 localStorage 读取，最多 5 篇）
- 每条显示：文件名 + 上传时间 + 对话条数
- 点击后提示「请重新上传该 PDF 文件以继续对话」（PDF 二进制不存储在 localStorage，只存对话文本）

### 3.1 技术栈选型

| 层级 | 技术 | 版本要求 | 说明 |

```

### 3.3 关键技术决策

#### 3.3.1 流式输出方案

- 后端使用 Next.js Route Handler 的 `Response` 对象配合 `ReadableStream` 转发 LLM 的 SSE 流
- 前端使用 `fetch` API + `response.body.getReader()` 消费流
- 逐 chunk 解析文本，追加到 React state

**后端流式转发核心逻辑（伪代码）：**
```typescript
// app/api/chat/route.ts
export async function POST(req: Request) {
  const body = await req.json();
  
  const llmResponse = await fetch(`${process.env.LLM_API_BASE_URL}/chat/completions`, {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.LLM_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: process.env.LLM_MODEL_ID,
      messages: body.messages,
      stream: true,
      // 思考强度参数（详见 3.3.2）
      ...getThinkingParams(),
    }),
  });

  // 直接将 LLM 的响应 body 流透传给前端
  return new Response(llmResponse.body, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
    },
  });
}
```

#### 3.3.3 PDF 文本提取

```typescript
// lib/pdfUtils.ts
import * as pdfjsLib from 'pdfjs-dist';

// 必须配置 worker
pdfjsLib.GlobalWorkerOptions.workerSrc = 
  `//cdnjs.cloudflare.com/ajax/libs/pdf.js/${pdfjsLib.version}/pdf.worker.min.js`;

export async function extractTextFromPDF(file: File): Promise<string> {
  const arrayBuffer = await file.arrayBuffer();
  const pdf = await pdfjsLib.getDocument({ data: arrayBuffer }).promise;
  
  let fullText = '';
  for (let i = 1; i <= pdf.numPages; i++) {
    const page = await pdf.getPage(i);
    const content = await page.getTextContent();
    const pageText = content.items
      .filter((item): item is TextItem => 'str' in item)
      .map(item => item.str)
      .join(' ');
    fullText += pageText + '\n\n';
  }
  
  // 截断至 60,000 字符
  if (fullText.length > 60000) {
    fullText = fullText.slice(0, 60000) + '\n\n[内容已截断，仅分析前 60,000 字符]';
  }
  
  return fullText;
}
```

## 五、后端 API 设计

### 5.1 `/api/polish` — 论文润色接口

**方法**：`POST`  
**描述**：接收用户文本和润色模式，调用 LLM 并以 SSE 流式返回结果

**请求体（Request Body）：**
```typescript
{
  text: string;           // 待润色文本，最大 10000 字符（服务端二次校验）
  mode: 'light' | 'academic' | 'rewrite';
  language: 'zh' | 'en' | 'auto';  // 前端检测到的语言
}
```

**响应（Response）：**
- `Content-Type: text/event-stream`
- 标准 OpenAI SSE 格式，每个 data chunk 包含 `delta.content` 字段
- 最终以 `data: [DONE]` 结尾

**服务端校验：**
- `text` 字段不为空且长度 ≤ 10000（超出返回 400）
- `mode` 字段枚举值校验

**错误响应格式：**
```json
{ "error": "错误说明文字", "code": "VALIDATION_ERROR" }
```

### 5.2 `/api/chat` — PDF 问答接口

**方法**：`POST`  
**描述**：接收完整对话历史和 PDF 文本内容，调用 LLM 并以 SSE 流式返回

**请求体：**
```typescript
{
  messages: Array<{
    role: 'user' | 'assistant';
    content: string;
  }>;
  pdfContent: string;    // PDF 提取的全文文本（最大 60000 字符）
}
```

**重要**：`pdfContent` 在后端被注入到 system message 中，不出现在 `messages` 数组里，前端每次请求都需要携带 `pdfContent`（无服务端 session 状态）

**响应**：与 `/api/polish` 相同（SSE 流）

**服务端校验：**
- `messages` 数组长度 ≤ 200（防止超长对话占用过多 token）
- `pdfContent` 长度 ≤ 60000

### 5.3 LLM 调用封装（`lib/llm.ts`）

```typescript
interface LLMCallOptions {
  messages: ChatMessage[];
  systemPrompt?: string;
  stream: true;
}

export async function callLLM(options: LLMCallOptions): Promise<Response> {
  const baseUrl = process.env.LLM_API_BASE_URL!;
  const apiKey = process.env.LLM_API_KEY!;
  const modelId = process.env.LLM_MODEL_ID!;

  const allMessages: ChatMessage[] = options.systemPrompt
    ? [{ role: 'system', content: options.systemPrompt }, ...options.messages]
    : options.messages;

  const response = await fetch(`${baseUrl}/chat/completions`, {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${apiKey}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: modelId,
      messages: allMessages,
      stream: true,
      max_tokens: 4096,
      ...getThinkingParams(),
    }),
  });

  if (!response.ok) {
    throw new Error(`LLM API error: ${response.status} ${response.statusText}`);
  }

  return response;
}
```

### 6.1 论文润色 System Prompt

```
你是一位专业的学术论文润色专家，精通中英文学术写作规范。

用户将提供一段论文文字，请按照以下要求进行润色，并**严格按照指定格式逐段输出**：

润色模式：{{MODE_DESCRIPTION}}

输出格式要求（必须严格遵守）：
每个自然段对应一个输出块，格式如下：

---PARA_START---
【原文】
（在此原样复制该段原文）

【问题分析】
（列举该段存在的具体问题，每条以"- "开头；若无问题，写"- 表达准确，无需修改"）

【润色结果】
（在此输出润色后的段落全文）
---PARA_END---

注意事项：
1. 每个段落对应一个 ---PARA_START--- 到 ---PARA_END--- 块，不可合并
2. 不要在块外输出任何额外文字（包括开头的问候语、结尾的总结语）
3. 论文语言为：{{LANGUAGE}}，请使用相同语言进行润色
4. 如遇专业术语，保留原术语不做翻译
```

**`{{MODE_DESCRIPTION}}` 对应值：**

| mode | 描述文字 |
|---|---|
| `light` | 轻度润色 — 保留原文结构与表达意图，仅修正语法错误、病句、逻辑不通处及用词不当，不改变句式结构 |
| `academic` | 学术加强 — 在保留原意的基础上，提升学术语气和专业性，规范术语使用，增强句间逻辑连接，去除口语化表达 |
| `rewrite` | 结构重写 — 在保留核心论点、数据和事实的前提下，可大幅重组行文逻辑、改写句式结构，以达到最佳学术表达效果 |

**`{{LANGUAGE}}` 对应值：**
- `zh` → `中文`
- `en` → `英文`
- `auto` → `请根据原文语言自动判断`

## 十二、Vercel 部署说明

### 12.1 部署步骤

1. 将项目代码推送到 GitHub 仓库
2. 登录 Vercel，点击「Import Project」，选择对应 GitHub 仓库
3. Framework Preset 选择 **Next.js**（应自动识别）
4. 在「Environment Variables」部分添加所有必要的环境变量（见第七章）
5. 点击「Deploy」

### 12.2 注意事项

- **Vercel Hobby 计划**：API Route 默认超时时间为 **10 秒**，对于长文本润色可能不够。建议在 `route.ts` 中配置：
  ```typescript
  export const maxDuration = 60; // 秒（Hobby 计划最大 60s，Pro 计划 300s）
  ```
- **Vercel 请求体大小限制**：默认 4.5MB，PDF 文本内容（60,000 字符约 60-180KB）不会触发限制
- **CORS**：API 路由仅供同域前端调用，无需额外配置 CORS

### 12.3 `.env.example` 文件内容（提交至 Git）

```bash
# 复制此文件为 .env.local 并填入实际值
LLM_API_BASE_URL=https://your-llm-api-endpoint.com/v1
LLM_API_KEY=your-nvidia-api-key-here
LLM_MODEL_ID=your-model-id-here
LLM_THINKING_INTENSITY=high
LLM_MAX_TOKENS=4096
```

## 十三、非功能性需求

### 13.1 性能目标

| 指标 | 目标值 |
|---|---|
| 首屏 FCP（First Contentful Paint） | < 1.5s |
| 页面切换响应 | < 200ms（客户端路由） |
| PDF 文本提取（10MB 文件） | < 5s |
| AI 首 token 响应（streaming 开始前） | < 5s（取决于模型） |
| Lighthouse 性能分数 | ≥ 85 |

### 13.2 浏览器兼容性

- Chrome 90+
- Firefox 88+
- Safari 14+
- Edge 90+
- **不支持 IE**（明确声明）

### 13.3 安全性注意事项

- API Key 仅存在于服务端环境变量，前端 bundle 中不包含任何密钥信息
- 输入文本在后端做长度校验，防止异常大请求
- 无用户数据上传至服务器，PDF 文本由前端提取后仅在当次 API 请求中传输，不存储于任何服务端

### 13.4 可访问性（Accessibility）

- 所有交互元素有明确的 `aria-label`
- 键盘导航支持（Tab 切换焦点，Enter/Space 激活按钮）
- 颜色对比度满足 WCAG 2.1 AA 标准（4.5:1）
- 图标类按钮必须有文字说明或 `aria-label`
